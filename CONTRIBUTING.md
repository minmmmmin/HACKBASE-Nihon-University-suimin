# 開発ルール

チーム開発をスムーズに進めるための最低限のルールです。

## 開発の流れ

1. **Issueを切る** — 作業を始める前に、やることをIssueにする（1タスク1Issue）
2. **ブランチを切る** — `main` から作業ブランチを作る（下の命名規則を参照）
3. **コミット** — こまめに、意味のある単位で
4. **PRを出す** — `main` へPull Requestを作成し、テンプレに沿って記入
5. **レビュー** — 1人以上のレビューを受けてからマージ
6. **マージ後** — ブランチは削除する

## ブランチ命名規則

`種別/issue番号` の形式にする。

```sample
feature/12
```

- Issue #12 に対応するブランチなら `feature/12`

## コミットメッセージ

内容を書いてください

```sample
- 店舗推薦APIのバリデーションを追加
- CORSの許可元を環境変数から取得するよう修正
- READMEにデプロイ手順を追記
```

## プルリクエスト

- タイトルは分かりやすく。関連Issueは本文に `close #番号` で紐付ける
- マージ前に必ずローカルで動作確認する:

```bash
npm run check   # Lint / Format / import整理
npm run build   # フロントの本番ビルド
npm run dev     # フロント(:5173) / バック(:3000) 起動確認
```

- `main` への直接pushは禁止。必ずPR経由でマージする

## コードスタイル

- フォーマット・Lintは **Biome** に統一（ESLint / Prettierは使わない）
- コミット前に `npm run check` で整形しておく

## やらないこと

- `.env` や APIキーをコミットしない
- 巨大なPRを作らない（レビューしやすい単位に分割する）
